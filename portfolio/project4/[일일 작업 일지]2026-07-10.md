## 2. 문제 발견 → 변경 → 성과

### 2-1. 역행적 정보 배치 (AutoAttack 위치)

**Before**
- `AutoAttack` 시퀀스 다이어그램이 3장(DLV) 내부에 위치
- DLV 개념을 설명하는 맥락에서 무기 공격 흐름이 선행 등장 → 논리 역행

**After**
- `AutoAttack` 시퀀스 다이어그램 및 설명을 `5.6절` (무기 시스템 → AutoAttack)으로 이동

**Why**
- 독자가 무기 구조(5.1~5.5)를 이해한 후 제어 흐름(5.6)을 보는 선형적 독해 흐름 확보

---

### 2-2. UI 정보 파편화 → 독립 장 통합

**Before**
- UI 관련 내용이 2장(기획) 말미와 7장(벡터 그래픽스)에 분산 배치

**After**
- 독립 장(8장)으로 통합: `8. UI 아키텍처 및 하이브리드 인터랙션`

**Why**
- 중앙 집중 UX, 네온 테마, 하이브리드 UI 발전 과정, Lerp 스케일링은 단일 설계 맥락으로 묶어야 가독성 확보

---

### 2-3. AudioPool 분류 오류 → 7장 공통 최적화로 이동

**Before**
- `5.7. AudioPool` → 무기 시스템(5장) 하위에 배치 (맥락 불일치)

**After**
- 5.7 제거 → `7.1. 공통 런타임 최적화 기법` 하위 `AudioPool` 소항목으로 통합
- Shapes GPU 인스턴싱과 함께 공통 최적화 묶음으로 구성

**Why**
- AudioPool은 무기의 기능이 아닌 씬 전체 사운드 최적화 기법. 무기 장과 분리 필요

---

### 2-4. 기술 서술 비일관성 → 코드 스니펫 추가

**Before**
- 4장(HexGrid)에만 코드 예시 존재
- 3장(DLV), 5장(무기 시스템)은 bullet 서술만 존재 → 기술 깊이 불균형

**After**
- 3장: `WeaponSO.cs (PureData)`, `RuntimeDataWeapon.cs` 코드 추가
- 5장: `Weapon.cs (추상 클래스)`, `ExplosiveWeapon.cs (자식 클래스)` 코드 추가

**Why**
- 포트폴리오 문서는 아키텍처 설명과 코드 예시가 함께 있어야 구현 능력 증명 가능

---

### 2-5. 전체 장 순서 재편 (문서 흐름 전면 개선)

**Before 순서**
```
1장 개요 → 2장 장르 기획 → 3장 DLV → 4장 HexGrid
→ 5장 무기 → 6장 고민과선택 → 7장 최적화(Shapes+AudioPool+CSV) → 8장 UI
```

**After 순서**
```
1장 개요 → 2장 장르 기획 → 3장 UI → 4장 DLV
→ 5장 고민과선택 → 6장 HexGrid → 7장 무기
→ 8장 공통 최적화(Shapes+AudioPool) → 9장 CSV Importer
```

**Why (항목별)**

| 변경 | 이유 |
|---|---|
| UI 3장 전진 | 장르 분석(2장) 직후 UX 설계 제시 → 기획 의도 → 시각 설계 → 기술 구조 자연스러운 흐름 |
| 고민과선택 5장 전진 | 구현 챕터(HexGrid/무기) 이전에 설계 결정 배경 제시 → 인과 흐름 정상화 |
| CSV Importer 9장 독립 | 런타임 최적화(8장)와 오프라인 에디터 파이프라인(9장) 명확히 분리 |
| 7장 이질성 해소 | 기존 7장에 Shapes(렌더링), AudioPool(사운드), CSV(에디터) 혼재 → 각 독립 챕터로 분화 |
| 4.1 소제목 신설 | DLV 3장만 `x.1.` 소제목 없이 bullet 나열 → 전체 구조 일관성 확보 |
