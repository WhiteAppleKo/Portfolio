* **비교 매핑**:
  * 👎 **아쉬운 예 (자의적 평가)**:
    * `Unity ★★★☆☆` / `C# ★★☆☆☆`
    * `Firebase (中)`
    * `"많이 다뤄보았습니다"` (신뢰도 0%)
  * 👍 **좋은 예 (구체적 경험 중심)**:
    * `Unity` — **FSM 유한상태머신 기반의 전투 및 몬스터 웨이브 시스템** 자체 설계/구현

    * `"낯선 외부 API를 설계 가이드에 맞춰 어떻게 분석하고 빠르게 내재화했는지"` 서술
* **왜**: 점수는 주관적이며 검증이 불가능합니다. 면접관은 기술 스택을 활용한 **실제 조율 역량**을 검토합니다.

## 📌 Slide 08: STAR 서사 구조 — 자소서 경험 작성법
* **타이틀**: 경험 서술의 정석 = **STAR**
* **구조 및 예시**:
  * **S (Situation · 상황)**: "팀 프로젝트 진행 중 Photon Fusion 네트워크 엔진을 처음 도입하게 됨."
  * **T (Task · 과제)**: "낯선 네트워크 동기화 개념과 복잡한 API 구조로 인해 프로젝트 핵심 기능 구현에 병목 발생."
  * **A (Action · 행동)**: "공식 문서 및 샘플 패키지 코드를 역공학 분해하여 최소 동작 예제를 빌드하고, 팀원들을 위한 동기화 아키텍처 가이드를 작성 공유함."
  * **R (Result · 결과)**: "네트워크 세션 동기화 완료 및 팀 전체의 멀티플레이 지연시간 최적화 달성."
* **교훈**: 문제 상황(T)에 대해 단순히 '어려웠다'로 서술을 끝내지 말고, 본인이 주체적으로 행한 **행동(A)과 교훈**을 기술적으로 드러내어 전문가의 성숙함을 증명하십시오.

## 📌 Slide 11: 기술 문서의 5단 구조 (Five Steps)
* **타이틀**: 기술 문서 작성의 **5단계 프레임워크**
* **5단 레이아웃 상세**:
  1. **① 개요 (What / Why)**:결론부터 제시하여 10초 이내에 독자가 문서의 타겟 문제와 가치를 알게 함.
  2. **② 시스템 구조 (Architecture)**: 구성요소 간의 관계 및 흐름을 클래스 다이어그램 등 다이어그램으로 시각화.
  3. **③ 핵심 기능 (Implementation)**: 핵심적인 런타임 코드 스니펫(50줄 이내)을 통해 구체적 사용법과 동작 원리 증명.
  4. **④ 고민과 선택 (Trade-offs)**: 정답 찾기가 아닌, 대안 간의 장단점 비교를 통해 이 결정을 내린 의사결정 논리 기술.
  5. **⑤ 결과 · 회고 (Retro)**: 수치 데이터를 통해 개선 결과를 증명하고, 남은 기술 부채 및 차기 보완 계획을 제시.
* **핵심 가치**: 기술 문서의 목적은 "내가 코딩했다"가 아니라, **"내 코드를 제3자에게 논리적으로 설명할 줄 안다"**는 엔지니어링 소통 능력을 보이는 것입니다.

## 📌 Slide 14: 예시: 핵심 기능 (RequestCraft) 소스 코드 명세
* **서브 타이틀**: ③ 핵심 기능: 어떻게 동작·사용하나
* **C# 핵심 코드 블록**:
```csharp
public void RequestCraft(string recipeId)
{
    var recipe = FindRecipe(recipeId);              // LINQ를 통한 레시피 데이터 조회
    if (recipe == null) { Publish(Fail); return; }

    // [소비 전 검증] -> 오작동으로 인한 재료 유실 방지
    if (!inventory.HasIngredients(recipe.Ingredients))
    { Publish(Fail); return; }

    // [roll 함수 의존 주입] -> 외부에서 난수 주입하여 결정론적 단위 테스트 환경 보장
    var success = roll() < recipe.SuccessRate;      
    inventory.Consume(recipe.Ingredients, success ? 1.0f : 0.5f); // 페널티 차등 적용
    if (success) inventory.Add(recipe.Result);

    Publish(success ? Success : Fail);              // 이벤트 버스 기반 전파
}
```
* **동작 원리 분석**:
  * **소비 전 검증**: 연산 진행 전 유효성 사전 검사로 예외 차단.
  * **roll 외부 주입**: 랜덤 시드를 주입 가능하게 설계하여 테스트 100% 검증 가능화.
  * **이벤트 발행**: `Publish()` 단일 지점으로 결과를 캡슐화해 UI 단과 느슨하게 연결.

## 📌 Slide 15: 예시: 고민과 선택 (Trade-offs) 비교표
* **서브 타이틀**: ④ 고민과 선택: 왜 이 방법인가
* **결과 전파 방식 대안 분석**:
  * **[대안 A] 직접 참조 (Direct Reference)**:
    * **장점**: 코드 직관적, 디버그 시 흐름 추적이 쉬움.
    * **단점**: 강한 결합 발생. 결과 연동 대상(예: 사운드, 도전과제)이 늘어날 때마다 Manager 소스 코드 수정 발생.
  * **[대안 B] 이벤트 전파 (Event Publication) — 선택 ✓**:
    * **장점**: 느슨한 결합 달성. 새로운 UI나 효과 컴포넌트를 코딩해도 Manager 측 수정이 전무함.
    * **단점**: 런타임 이벤트 호출이 많아지면 디버깅 및 실행 흐름 추적 피로도 상승 (단점 감수).
* **Rationale**: "결정론적 B안 선택. 향후 확장성을 확보하기 위해 디버깅 흐름 추적 비용 상승이라는 단점을 기꺼이 감수하고 의존성 제거를 지향함."

## 📌 Slide 18: 레이아웃 3대 원칙
* **메인 타이틀**: 레이아웃이 전달력을 결정합니다.
* **3대 설계 규칙 swatches**:
  1. **한 화면 한 메시지**: 하나의 슬라이드에는 오직 하나의 주제만 담고, 정보가 많아지면 과감히 페이지를 나눔.

  3. **시각적 위계 (Hierarchy)**: 폰트 크기, 색상 대비, 선 굵기를 활용해 타이틀, 본문, 강조 단락, 캡션의 위계를 계층 설계.
* **비유**: **도로 표지판** — 운전 중 0.5초 이내에 직관 인지되어야 하는 도로 표지판처럼, 슬라이드를 보자마자 결론이 인식되도록 Z자 구조로 핵심 요소를 배치하십시오.
* **👎 아쉬운 구조**: 정렬 안 됨, 한 장에 3대 주제 혼용, 크기 대비 부족으로 인한 가독성 저하.

## 📌 Slide 27: Mermaid 핵심 기호 치트시트
* **타이틀**: 핵심 기호 한 장 정리
* **다이어그램별 문법 핵심 요약**:
  1. **클래스 다이어그램 (Class)**:
     * `<|..` : 인터페이스 구현 (Implementation)
     * `-->` : 소유/참조 (Association)
     * `..>` : 의존 (DIP, Dependency)
     * `+` / `-` : public / private 접근 제한자
  2. **시퀀스 다이어그램 (Sequence)**:
     * `->>` : 동기식 메시지 전달 (Call)
     * `-->>` : 비동기식 결과 반환 또는 이벤트 전파 (Return)
     * `Note over [객체]` : 설명 주석 추가
  3. **상태 다이어그램 (State)**:
     * `[*]` : 시작점 및 소멸점 정의
     * `-->` : 상태 전이선
     * `: [조건]` : 전이를 일으키는 트리거/조건 기술
  4. **플로우차트 (Flowchart)**:
     * `([ ])` : 시작/종료 타원형 기호
     * `{ }` : 조건 판단 마름모 기호
     * `[ ]` : 일반 처리 프로세스 사각형 기호
     * `-->|라벨|` : 분기 화살표 경로 설명
  5. **ER 다이어그램 (ERD)**:
     * `||--o{` : 1 대 0..N 관계 매핑
     * `||--|{` : 1 대 1..N 관계 매핑

* **4대 요소 구성**:
  * **역할 (Role)**: AI의 전문 페르소나 설정 ("너는 테크니컬 라이팅 전문가야")
  * **맥락 (Context)**: 이 문서를 읽을 타겟 독자와 목적 전달 ("유니티 신입 이력서용 기술 명서야")
  * **제약 (Constraint)**: 줄수, 정렬, 색상수 한정 ("코드는 50줄 이내, 다크 코드 스Switches")
  * **형식 (Format)**: 결과물의 최종 반환 포맷 ("Marp 마크다운 코드블록으로만 대답해줘")
* **비유**: **외주 발주서** — 계약서 조건이 명확하고 세부적이어야 원하는 퀄리티의 제품을 수령할 수 있는 이치와 동일합니다.
