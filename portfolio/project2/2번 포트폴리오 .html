1. 개발 환경 설정 & 코드 컨벤션 정립

팀 단위 협업 작업이 처음이었던 개발진 간의 협업 마찰 및 깃(Git) 브랜치 병합 시 코드 컨플릭트를 최소화하기 위해 공통 코딩 스탠다드를 명문화하고 Notion을 통해 정기 배포했습니다.

  • 명명 규칙: 클래스 PascalCase, 지역변수 camelCase, private 멤버변수 m_PascalName, bool 변수 bPrefix를 엄격히 지정하여 통일성 확보.
  • Rider IDE 서식 통일: Rider 설정 파일을 XML 서식 형태로 버전 관리 배포하여 개발 에디터 환경 내에서 코드 포맷이 자동으로 동기 정렬되도록 연동.

2. 인터페이스 기반 협업 프로세스 도입

기존의 너무 세밀하게 쪼개진 태스크 일정 수립 방식은 상호 결합도가 높은 기능들 간의 구현 대기 병목을 유발해 전체 개발 템포를 저하시켰습니다. 이를 개선하기 위해 인터페이스 기반 분리 개발 패턴을 설계하여 협업 생산성을 높였습니다.

인터페이스와 목(Mock) 객체의 연동 Rationale
플레이어 구현 파트와 게임 심판 로직 파트가 서로의 실물 코드가 작성되기까지 대기할 필요가 없도록 상호 인터페이스(IPlayerActions, IGameReferee)를 최우선 확정했습니다. 이 인터페이스 규격을 통해 각각 Mock 플레이어 및 Mock 심판 객체를 생성하여 자기 모듈의 논리적 무결성을 독립 개발 및 선(先) 단위 검증을 통과시킬 수 있었습니다.

3. Photon Fusion 2 네트워크 구조 설계

3.1. 공유 모드 (Shared Mode) 선정 이유

본 보드게임의 핵심적인 네트워킹 리스크는 방장(Master Client)의 예기치 않은 세션 중단(이탈) 시 게임 룸 전체가 폭파되어 다른 잔여 플레이어들의 플레이가 불능에 빠지는 현상이었습니다. 이 리스크를 방어하기 위해 다음과 같은 Rationale로 연결 형식을 제안했습니다.

  • 공유 모드(Shared Mode) 채택: 일반적인 Host Mode와 달리 룸 상태(State Authority)가 클라이언트별 객체 단위로 유연하게 위임 분산되므로, 방장이 세션 단절로 탈퇴해도 룸이 소멸되지 않고 방 권한만 즉시 다른 클라이언트에 이양되어 게임이 끊김 없이 지속 유지됩니다.

3.2. 네트워크 변수(Networked Properties) 단순화

복잡한 동기화 정보 전송을 위해 커스텀 직렬화 구조체를 설계하려 했으나, 런타임 통신 빈도 및 파라미터 양을 분석한 결과 최적화를 도출했습니다. Photon의 [Networked] 식별자가 적용된 기본 변수 동기화 구조로 단일화 처리하여 전송 대역폭을 최소화하고 틱 레이턴시를 대폭 절감했습니다.

4. GameManager 및 RuleManager의 역할 분리

유니티 싱글톤 매니저 객체인 GameManager에 모든 게임 진행 스택 및 보드판 매칭 규칙 검사 로직이 병합되는 스파게티 설계를 방지하고자 심판 역할을 서브 컴포넌트로 디커플링했습니다.

  • GameManager: 인게임 상태 중앙 관리 및 동기화, UI 연계, 턴 제어의 흐름 조율 담당 (싱글톤 생명주기).
  • RuleManager: 플레이어가 턴 중에 선택하고 매칭한 타일 데이터의 유효성을 실시간 판별하고 검사하는 고유의 심판 판정 기능에만 책임을 제한.

5. 플레이어 실시간 이탈 예외 제어 핸들러

게임 실행 도중 클라이언트 한 명이 연결 해제되었을 때, 남은 플레이어들의 게임 플레이 안전성을 보장하기 위해 다음과 같은 이탈 예외 제어 방어 코드를 적용했습니다.

public override void OnPlayerLeft(NetworkRunner runner, PlayerRef player)
{
    // 1. 이탈한 플레이어의 캐릭터를 찾아 반투명 머티리얼로 시각 변경 처리
    var leftPlayerObj = GetPlayerObject(player);
    leftPlayerObj.SetTransparentMode(true);

    // 2. 턴 스택 제어 리스트에서 해당 플레이어를 완전히 배제
    m_activePlayers.Remove(player);

    // 3. 만약 이탈한 유저가 '현재 턴을 소유하고 진행 중인 플레이어'였을 경우
    if (m_currentTurnPlayer == player)
    {
        // 턴 중단 및 즉시 턴 인덱스를 다음 살아있는 플레이어에게 강제 이탈 위임
        ForcePassNextTurn();
    }
}
